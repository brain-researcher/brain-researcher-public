# Behavior Task Generation (psyflow)

Canonical runbook for generating behavior task scaffolds from natural-language
requirements, validating them against the psyflow runtime, and ingesting
executed runs back into the shared `BehaviorTrial` pipeline.

## Install

The psyflow stack is an optional extra because it pulls a git-pinned runtime:

```bash
pip install -e ".[behavior-task]"
# which resolves:
#   psyflow @ git+https://github.com/TaskBeacon/psyflow.git@cb2773299e178f967380cc0f9520c056810cfd2a
#   PyYAML>=6.0
```

Core `brain_researcher` imports (and unit tests) work without the extra. All
psyflow access is gated through `psyflow_adapter._import_psyflow` and raises
`PsyflowNotInstalledError` when unavailable.

## Pipeline

```
prompt
  └─► behavior.paradigm_planner            (existing)
        └─► behavior.resolve_task_spec     (paradigm defaults + digest)
              └─► behavior-review-v1       (spec-digest-bound approval gate)
                    └─► behavior.generate_psyflow_task   (writes <out>/planned/…)
                          ├─► behavior.validate_task_spec (optional re-check)
                          ├─► psyflow.validate          (optional; skipped w/o extra)
                          ├─► [external execution]      (psyflow / PsychoPy / human)
                          └─► behavior.ingest_psyflow_run (reads <out>/run/…)
                                └─► behavior.qc_scan    (existing)
                                └─► behavior.export_bids (existing)
```

## Directory layout

```
<out>/
  planned/
    <paradigm>/          # generated by behavior.generate_psyflow_task
      config/config.yaml
      main.py
      task_spec.json
      spec_digest.txt
  run/
    <session_id>/        # populated by external execution
      …CSV/JSON trial data…
```

`planned/` is authored by this tool; `run/` is authored by psyflow/PsychoPy/etc.
The adapter enforces this split — `ingest_psyflow_run` rejects a `run_data_dir`
that is not under `<out>/run/`, and `write_psyflow_scaffold` refuses to write
anywhere under `<out>/run/`.

## Tool reference

| Tool | Role | Safety |
|------|------|--------|
| `behavior.resolve_task_spec` | Paradigm defaults → `behavior-task-spec-v1` + digest | safe |
| `behavior.validate_task_spec` | Validate a spec payload and return digest | safe |
| `behavior.generate_psyflow_task` | Writes planned scaffold (approval gate) | safe |
| `behavior.ingest_psyflow_run` | Normalize psyflow run under `<out>/run/` | safe |

Existing tools (unchanged): `behavior.paradigm_planner`, `behavior.ingest_taps`,
`behavior.qc_scan`, `behavior.export_bids`.

## Approval gate

The generate step is **spec-digest-bound**. Callers submit a
`behavior-review-v1` payload with `approved=True` and
`spec_digest == sha256(model_dump(mode="json", sort_keys, separators=(",",":")))`.
Mismatches or unapproved reviews return `approval_gate_failed` without writing
any files.

## Marimo notebook entrypoint

A guided, human-in-the-loop version of this pipeline lives in the
`behavior_task_builder` marimo notebook template. It drives the same five
tools — `behavior.paradigm_planner`, `behavior.resolve_task_spec`,
`behavior.validate_task_spec`, `behavior.generate_psyflow_task`,
`behavior.ingest_psyflow_run` — and enforces the same digest-bound approval
gate (resolve → view digest → approve → generate; editing the spec after
approval re-invalidates the digest by design).

Inside hosted marimo, this template uses a dedicated local stdio BR client
instead of the remote MCP HTTP target. That is intentional: generated
`planned/` scaffolds and later `run/` ingest paths must resolve inside the
notebook workspace, not the shared MCP service filesystem.

Open it with:

```bash
br notebook open behavior_task_builder --port 2718
```

The `[behavior-task]` extra above is required for `generate_psyflow_task`
and `validate_task_spec` to exercise psyflow; without it the notebook still
resolves specs and emits digests but skips the psyflow-backed steps. The
marimo user guide (`docs/user-guide/marimo.md`) links back here for the full
contract reference.

## Example

```python
from pathlib import Path
from brain_researcher.behavior.catalog import resolve_defaults
from brain_researcher.behavior.task_spec import spec_digest
from brain_researcher.services.tools.behavior_tools import (
    BehaviorGeneratePsyflowTaskTool,
)

spec = resolve_defaults("n_back", {"n_trials": 120})
digest = spec_digest(spec)

tool = BehaviorGeneratePsyflowTaskTool()
result = tool._run(
    spec=spec.model_dump(mode="json"),
    out_dir=str(Path("./behavior_out")),
    review={"spec_digest": digest, "approved": True, "reviewer": "alice"},
)
print(result.status, result.data["bundle"]["planned_dir"])
```

## Not in scope (v1)

- PsychoJS / browser-side task hosting.
- In-scanner trigger orchestration (scanner knobs are captured in
  `ScannerProfile` but not actuated).
- Online stimulus generation beyond static psyflow config.
- Neurogym RL tasks (different audience — RL agents vs human subjects).

These belong to separate workflows and are intentionally excluded to keep the
`behavior-task` extra narrow.
