# BR-KG Search & Evidence Architecture — Spec v2.1 (Engineering)

## 0. Goals and Scope

**Goals**
- Provide traceable evidence retrieval separate from multiverse planning.
- Control scoring decay by `doc_role`, not by physical year shards.
- Improve recall via KG aliasing and fulltext search.

**Non-Goals**
- Do not generate or rewrite specs in the Evidence layer.
- Do not treat derived multiverse candidates as literature-backed evidence.

---

## 1. Physical Storage and Metadata

### 1.1 Physical GFS Stores (Governance-Oriented)

- **Store A: `br-kg-living-tooling`**
  - Content: 2020+ tool docs, release notes, BIDS specs, best practices.
  - Policy: aggressive updates, replace older versions.

- **Store B: `br-kg-modern-literature`**
  - Content: 2010–2019 empirical studies and method reviews.
  - Policy: append-only (incremental).

- **Store C: `br-kg-canonical-anchors`**
  - Content: canonical methods/statistics (Friston, Cox, Worsley, etc.).
  - Policy: allowlist, very stable.

### 1.2 Metadata Schema (Required Fields)

```json
{
  "doc_id": "...",
  "store_id": "br-kg-modern-literature",
  "doc_role": "empirical",
  "title": "...",
  "published_year": 2016,
  "updated_at": "2024-10-12",
  "tool_name": "fmriprep",
  "tool_version": "24.0.1",
  "aliases": ["ICA-AROMA", "AROMA"]
}
```

**Rules**
- Default `doc_role` is `empirical` when classification fails.
- `doc_role` can be overridden via a allowlist for stability.
- For tooling documents, `updated_at`/`tool_version` should drive freshness.

### 1.3 Auto-Tagging Rules (Heuristics)

**Rule 1: Tooling**
- If filename/title contains any of: `fmriprep`, `fitlins`, `bids`, `docker`, `release notes`
- Or content matches version pattern like `v\d+\.\d+\.\d+`
- Then `doc_role = tooling_spec`

**Rule 2: Guideline**
- If title contains any of: `best practice`, `guideline`, `standard`, `consensus`, `cobidas`
- Then `doc_role = guideline`

**Rule 3: Foundation**
- If author is allowlisted (e.g., Friston, Worsley, Cox) AND `published_year < 2010`
- Then `doc_role = foundation`

**Fallback**
- Else `doc_role = empirical`

## 2. KG Recall Layer (Neo4j)

### 2.1 Alias Augmentation

Add `aliases` to:
- `TaskSpec`
- `Construct`
- `Dataset`

Aliases can be generated by rules and manual curation (e.g., `confounds_aroma` -> `ICA-AROMA`, `AROMA`).

### 2.2 Fulltext Index

```cypher
CREATE FULLTEXT INDEX neuro_fulltext
FOR (n:TaskSpec|Dataset|Construct)
ON EACH [n.name, n.description, n.aliases]
```

## 3. Evidence Orchestration Layer

### 3.1 Retrieval Pipeline

1. KG Recall via fuzzy fulltext index.
2. GFS Retrieval across all stores.
3. Evidence binding (alias + context filter).
4. Rerank (RRF + role decay + intent weights).

### 3.2 Evidence Binding (MVP)

**Alias classes**
- **Strong aliases** (direct method names like `ICA-AROMA`, `aCompCor`, `DVARS`): match is sufficient.
- **Weak aliases** (generic tokens like `motion`, `filter`): require context.

**Context keyword filter**
- Positive: `recommend`, `default`, `we used`, `pipeline`, `parameter`, `set to`, `chosen`, `implemented`
- Negative: `not used`, `limitation`, `future work`, `unrelated`

### 3.3 Performance Constraints (Binding Scope)

- Only bind evidence for the top 20–30 KG candidates by recall score.
- Avoid full-graph binding per query.
- Optional: precompile a single regex for strong aliases to scan snippets in one pass.

---

## 4. Scoring and Rerank

### 4.1 Multi-Store Merge: Global Normalized Score Sum

RRF discards score magnitudes and can nullify role/time decay. Instead, use
score normalization per store and apply decay at the item level.

1. Retrieve raw GFS scores per store.
2. Normalize scores within each store to 0–1.
3. Apply role/time decay per document:

```text
doc_score = norm_gfs_score * decay_factor(doc_role, year_or_updated_at)
```

4. Merge by `doc_score` and truncate.

### 4.2 Role-Based Decay

- `foundation`: no decay
- `guideline`: half-life 3–5y
- `tooling_spec`: half-life 1–2y (prefer `updated_at`)
- `empirical`: default decay

### 4.3 Intent Weighting

If query intent includes tool/pipeline/version, boost `tooling_spec` evidence.

- Use `?format=list` (or body `{format:"list"}`) to return the legacy list-only response.

## 6. Governance and Ops

- Enforce `doc_role` allowlist for stability.
- Support force-delete (cascade) for tooling store replacements.
- Track `evidence_status` to avoid user confusion in partial data states.

---

## 7. Execution Roadmap

**Phase 0 — Governance**
- Define `doc_role` rules and required metadata fields.
- Implement replace/delete strategy for tooling docs.

**Phase 1 — Recall**
- Add aliases to KG nodes.
- Create and validate fulltext index.

**Phase 2 — Orchestrator MVP**
- Implement KG+GFS retrieval and binding.
- Rerank with RRF and role decay.

**Phase 3 — Scale Up**
- Multi-store ingestion.
- Tune decay and intent weighting.

**Phase 4 — Multiverse Expansion**
- Priors-driven candidate generation.
- Option-level evidence attribution (no spec-level evidence claims).
